## 1. 버전 관리(Version Control) 이해

### 1.1 버전 관리의 필요성과 개념
* [cite_start]**필요성**: 시간 흐름에 따라 파일 집합의 변경 사항을 추적하여, 과거 어느 시점의 파일이나 소스 내용을 확인하고 그 상태로 되돌리고 싶을 때 사용됩니다[cite: 1472, 1868].
* [cite_start]**개념**: **시간의 흐름에 따라 파일 집합에 대한 변경 사항을 추적하고 관리**하는 것입니다[cite: 1540, 1866]. [cite_start]필요하면 과거 어느 시점으로 이동할 수 있는데, 이는 마치 “시간 여행”과 같습니다[cite: 1549].

### 1.2 커밋(Commit) 및 HEAD
* [cite_start]**저장소(Git Repository)**: 파일이나 폴더, 그리고 버전 관리를 위한 관련 파일을 저장해 두는 곳입니다[cite: 1749, 1879].
* [cite_start]**커밋(Commit)**: 버전 관리에서 **저장소의 현 상태를 저장하는 행위**이자 명령어입니다[cite: 1677, 1679, 1680, 1876]. [cite_start]이는 현 상태를 담는 스냅샷(사진)을 찍는 것에 비유됩니다[cite: 1678].
* [cite_start]**HEAD**: 가장 최근의 커밋을 가리키는 **포인터**입니다[cite: 1723, 1877].

## 2. 깃(Git) 개요 및 설치

### 2.1 깃(Git) 소개
* [cite_start]**개발**: **리누스 토발즈(Linus Torvalds)**가 2005년에 개발했습니다[cite: 2614].
* [cite_start]**정의**: 컴퓨터 파일의 변경을 추적하는 데 사용되는 **버전 관리 시스템**입니다[cite: 2164, 2649].
* [cite_start]**특징**: 대표적인 **오픈소스 소프트웨어** 프로젝트이며, 소스 코드 관리에 **분산 버전 제어 도구**가 사용됩니다[cite: 2189, 2192, 2653].
* [cite_start]**기능**: 여러 개발자가 함께 작업하며 소스 코드의 변경 사항을 추적하는 데 사용되며, 여러 개의 평행 분기를 통해 **비선형 개발**을 지원합니다[cite: 2651, 2652, 2654].

### 2.2 깃 설치 시 제공되는 소프트웨어
* [cite_start]**Git Bash**: **CLI(Command Line Interface)** 방식의 명령행 인터페이스를 제공합니다[cite: 1556, 2663, 2664].
* [cite_start]**Git GUI**: **GUI(Graphical user interface)** 방식의 인터페이스를 제공합니다[cite: 1583, 1585, 2685].

## 3. 깃의 내부 주요 영역 및 명령

### 3.1 깃 내부 저장소의 주요 영역
| 영역 명칭 | 별칭 | 주요 상태 |
| :--- | :--- | :--- |
| **작업 디렉토리** (Working Directory) | [cite_start]Working folder, Working tree, Work space [cite: 1871, 2705, 2706, 2848] [cite_start]| modified, untracked [cite: 2707] |
| **스테이징 영역** (Staging Area) | [cite_start]Index, Stage area [cite: 2708, 2849] [cite_start]| staged, indexed [cite: 2708] |
| **깃 저장소** (Git Repository) | [cite_start]Repository, .git directory [cite: 2709, 2850] | [cite_start]Committed [cite: 2709] |
| **임시 저장소** (Stash) | [cite_start]- [cite: 2710] [cite_start]| stash [cite: 2711, 2851] |

### 3.2 상태 간의 이동 명령 (지역 저장소 기준)
* [cite_start]**`add` 명령**: **작업 디렉토리**에서 **스테이징 영역**으로 변경 사항을 이동시킵니다[cite: 2595, 2719, 2853].
* [cite_start]**`commit` 명령**: **스테이징 영역**에서 **깃 저장소**로 변경 내용을 기록합니다[cite: 2595, 2720, 2855].

### 3.3 원격/지역 저장소 간 이동 명령
* [cite_start]**원격 저장소(Remote Repository)**: 원격 저장소 전용 서버에서 관리되며 여러 사람이 함께 공유하기 위한 저장소입니다[cite: 1765, 1766].
* [cite_start]**지역 저장소(Local Repository)**: 내 PC에 파일이 저장되는 개인 전용 저장소입니다[cite: 1767, 1768].
* [cite_start]**`Clone`**: 서버의 원격 저장소를 지역 저장소에 **복제**합니다[cite: 1776, 1777].
* [cite_start]**`Push`**: 지역 저장소의 내용을 **원격 저장소로 올립니다** (업로드)[cite: 1778, 1779, 1833, 1835].
* [cite_start]**`Pull`**: 원격 저장소에서 여러 사람이 작업한 파일을 **지역 저장소로 가져옵니다** (내리기)[cite: 1780, 1781, 1836, 1838].

## 4. 브랜치(Branch) 이해

* [cite_start]**개념**: 버전 관리를 수행하던 일련의 파일 집합을 통째로 복사해 **독립적으로 다시 개발을 진행하는 개념**입니다[cite: 2596, 2757, 2861, 2862].
* [cite_start]**역할**: 새로운 수정을 할 수 있는 또 다른 버전의 작업 흐름(workflow)을 만드는 데 사용되며, 프로젝트의 **비선형 개발**을 가능하게 합니다[cite: 2654, 2755].

## 5. 깃허브(GitHub) 개요

* [cite_start]**정의**: **깃 기반의 저장소 및 소프트웨어 협업 개발을 위한 웹호스팅 서비스**입니다[cite: 2166, 2219, 2262, 2775, 2791]. [cite_start]또한, **버전 관리를 위한 서버 저장소 및 프로젝트 개발을 위한 협업 관리 서비스**입니다[cite: 2864].
* [cite_start]**역할**: 전 세계 개발자가 함께 개발할 수 있도록 지원하는 **세계 최대의 오픈소스 공유 플랫폼**입니다[cite: 2809, 2827].
* [cite_start]**소유**: 2018년 6월 4일에 **마이크로소프트사**가 75억 달러(약 9조 원)에 인수했습니다[cite: 2267, 2268, 2575, 2829].
* [cite_start]**주요 기능**: 코드 호스팅 및 검토, 프로젝트 관리, 이슈, 레이블, 마일스톤, **끌어오기 요청(PR: Pull Request)** 등을 제공하여 협업을 지원합니다[cite: 2397, 2405, 2406, 2407].
